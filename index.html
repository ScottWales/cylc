Cylc

Forecast Suite Metascheduler

Cylc ("silk") is an advanced metascheduler that makes it easy to construct and control powerful, portable, suites for cycling weather and environmental forecasting systems.

.

Provenance

Cylc is developed by Dr Hilary James Oliver at NIWA (the National Institute of Water and Atmospheric Research, New Zealand). It was first prototyped in 2008, has been used to run NIWA's operational forecasting systems since 2010, and in 2011 was released as free open source software under the terms of the GNU General Public License.

Overview

Cylc decides when members of a suite of inter-dependent cycling tasks are ready to be submitted to a scheduler to be queued for execution.

Dependence between forecast cycles can usually be ignored in real time operation, but this places severe restrictions on catchup/delayed operation when it should be possible to run tasks from multiple cycles at once. Even simple suites like this one become complicated like this when intercycle dependence is taken into account (a, b, and c are warm-cycled forecast models).

Cylc's self-organizing scheduling algorithm explicitly manages dependencies between as well as within forecast cycles, allowing it to optimally interleave tasks from multiple cycles without dependency violations, potentially achieving very fast catchup from delays even in resource limited operations with little down time between cycles - as shown in this job schedule (cylc is above the time line).

Cylc also has very flexible and adaptive run time behaviour. Suites organically transition between delayed and real time operation as required; they adapt to insertion and removal of tasks; and they flow around failed or delayed tasks while problems are addressed - after which the delayed tasks catch up as quickly as possible.

See Notable Features below, and the introductory section of the User Guide for a more detailed discussion.

Requirements

  • Operating System: Unix or Linux
  • The Python programming language
  • Pyro 3 (Python Remote Objects)
  • Graphviz (graph layout engine)
  • Pygraphviz (a Python graphviz wrapper)

Notable Features

  • Tasks can be distributed across multiple heterogeneous hosts, with multiple job submission methods.
  • Most existing models and processing jobs can be used without modification as cylc tasks.
  • Simple, structured, intuitive suite definition: "draw" a dependency graph in concise textual form, and configure the execution environment and command line for each task.
  • Suite definitions are automatically validated against a comprehensive spec file, to catch most errors prior to run time.
  • Self-documenting unified command line interface covering all functionality from suite editing, searching, graphing, and validation; to interrogation and control.
  • Graphical interfaces covering the same, including a graph-based suite control interface. See screen shots.
  • Cryptographic suite security to prevent users from intervening in the operation of suites owned by others.
  • Run single tasks exactly as the suite would run them (job submission method and execution environment)
  • A network lockserver prevents accidental invocation of multiple instances of the same suite or task at the same time.
  • Suite databases enable sharing of suites among users.
  • Insertion and deletion of tasks at run time
  • Explicit handling of cold start processing to bootstrap warm-cycling suites from scratch.
  • Suites can be cold started, warm started, or restarted from a previous state.
  • Suites can be stopped and restarted in any state of operation.
  • Restart states are automtically dumped prior to actioning any intervention in a running suite.
  • In-suite spin-up processing via insertion of temporary tasks and addition of temporary prequisites on the fly.
  • Recursive purge: remove an entire sub-tree of tasks through to some future cycle time.
  • Conditional prerequisites
  • Task families
  • Task retry for automated failure recovery
  • Central alerting and timeouts for operational suites
  • Portable suites: just copy and run
  • Simulation mode: get the scheduling right, on an accelerated clock, without running real tasks
