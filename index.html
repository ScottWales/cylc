Cylc

Forecast Suite Metascheduler

Cylc (pronounced silk) is an advanced metascheduler for cycling weather and environmental forecasting systems.

Cylc makes it easy to construct and control powerful, portable, suites for numerical forecasting and related processing. It has been developed by Dr Hilary James Oliver at NIWA (the National Institute of Water and Atmospheric Research, New Zealand) since 2008, and was released as free open source software under the terms of the GNU GPL in 2011.

Overview

Cylc decides when members of a suite of interdependent cycling tasks are ready to be submitted to a scheduler to be queued for execution. Its self-organising scheduling algorithm, which explicitly manages dependencies between as well as within forecast cycles, allows it to optimally interleave tasks from multiple cycles without violating dependencies, potentially achieving very fast catchup from delays even in resource limited operations with little down time between cycles. Cylc suites organically transition between delayed and real time operation as required; they adapt to insertion and removal of tasks at run time; and they flow around failed or delayed tasks while problems are addressed - after which the delayed tasks catch up as quickly as possible.

An Illustration

Intercycle dependencies can usually be ignored in real time operation, but this places severe restrictions on what can be done in catchup/delayed (or case study) operation. Requirements

  • Operating System: Unix or Linux
  • The Python programming language
  • Pyro 3 (Python Remote Objects)
  • Graphviz (graph layout engine)
  • Pygraphviz (a Python graphviz wrapper)

Notable Features

  • Tasks can be distributed across multiple heterogeneous hosts, with multiple job submission methods.
  • Most existing models and processing jobs can be used without modification as cylc tasks.
  • Simple, structured, intuitive suite definition: "draw" a dependency graph in concise textual form, and configure the execution environment and command line for each task.
  • Suite definitions are automatically validated against a comprehensive specification.
  • Self-documenting unified command line interface covering all functionality from suite editing, searching, graphing, and validation; to interrogation and suite control.
  • Graphical interfaces covering the same, including a graph-based suite control interface. See screen shots.
  • Crypographic suite security - prevents users from accidentally or maliciously intervening in the operation of suite's owned by others.
  • Run single tasks exactly as the suite would run them
  • A network lockserver to prevent accidental invocation of multiple instances of the same suite or task at the same time.
  • Suite databases to enable sharing of suites among users.
  • Suites can be stopped and restarted in any state of operation.
  • Insertion and deletion of tasks at run time
  • Recursive purge: remove an entire sub-tree of tasks through to some future cycle time.
  • Conditional prerequisites
  • Task families
  • Central alerting and timeouts for operational suites
  • Portable suites: copy and run
  • Simulation mode: get the scheduling right, on an accelerated clock, without running the real tasks
