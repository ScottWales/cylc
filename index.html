Cylc

Forecast Suite Metascheduler

Cylc ("silk") is an advanced metascheduler that makes it easy to construct and control flexible, portable suites for cycling weather and environmental forecasting systems, and related data processing. Cylc suites optimally interleave tasks from multiple forecast cycles for fast catch up from delays, fast case studies, and fast offset parallel test suites.

.

Cylc is developed by Dr Hilary James Oliver at NIWA (National Institute of Water & Atmospheric Research, New Zealand). It was prototyped in 2008, has controlled NIWA's multi-model operational forecasting systems since 2010, and was released under the GNU General Public License in 2011.

Overview

A metascheduler, in this context, is a control system that determines when members of a suite of interdependent tasks are ready to execute (usually by submission to a batch queue scheduler). Cylc does this for suites of cycling tasks.

Dependence between tasks in different forecast cycles can be ignored in real time operation, but this severely restricts scheduling after delays when it should be possible to run tasks from multiple cycles at once. Even simple suites appear complicated when intercycle dependence is taken into account (a, b, and c represent warm-cycled forecast models). Cylc's self-organising scheduling algorithm solves this problem, and as a result it can optimally interleave tasks from multiple forecast cycles whenever the opportunity arises, potentially achieving very fast catch up from delays (cylc on top) even in resource limited operations with little down time between cycles. The same goes for processing historical data (cylc on top) or running parallel test suites that start behind the main operation.

Cylc also has very flexible, adaptive run time behaviour: suites naturally transition between delayed and real time operation as required; they adapt to insertion and removal of tasks; and they flow around failed or delayed tasks (which catch up as quickly as possible once the problem is addressed).

Cylc's suite design paradigm is simple and intuitive: "draw" a dependency graph in concise textual form, then configure the execution environment and command line for each task. Other Notable Features

First, see the Overview above.

  • Cylc can handle many hundreds of tasks in a single suite (see performance).
  • Tasks can be distributed across multiple hosts on a heterogeneous network, with multiple job submission methods.
  • Most existing models and processing jobs can be used without modification as cylc tasks; some require minimal modification.
  • Suite definitions are automatically validated against a comprehensive spec file, to catch most errors prior to run time.
  • Self-documenting command line interface covering all functionality from suite editing to suite control.
  • Graphical interfaces covering the same, including a graph-based suite control interface. See screen shots.
  • Cryptographic suite security: users cannot intervene in the operation of suites owned by others.
  • Run single tasks exactly as the suite would run them (job submission method and execution environment).
  • Network lockservers prevent accidental invocation of multiple instances of the same suite or task at the same time.
  • Suite databases enable sharing of suites among users.
  • Insertion and deletion of tasks at run time.
  • Explicit cold start processing to bootstrap warm-cycling suites from scratch.
  • Suites can be cold started, warm started, or restarted from a previous state.
  • Suites can be stopped and restarted in any state of operation.
  • Restart states are automatically dumped prior to actioning any intervention in a running suite.
  • Spin-up processing by insertion of temporary tasks and addition of new prerequisites on the fly.
  • Recursive purge: remove an entire sub-tree of tasks through to some future cycle time.
  • Conditional prerequisites (triggers).
  • Task families.
  • Task retry for automated failure recovery.
  • Central alerting and timeouts for operational suites.
  • Portable suites are easy to construct - just copy another user's suite and run it.
  • Simulation mode: get the scheduling right, on an accelerated clock, without running real tasks.

Performance

As of version 3.2.0 cylc can easily manage 1000 tasks in single suite. Beyond several hundred tasks the suite control GUIs may struggle - they have not yet been optimized for huge suites; for now you can run very large suites using cylc command line control and monitoring tools.

Requirements

  • Operating System: Linux or Unix 1
  • Python Version: 2.4 or later, but not Python 3.x as yet. 2
  • PyGTK, a Python wrapper for the GTK+ graphical user interface toolkit, included in most Linux Distributions.
  • Pyro 3 (Python Remote Objects) - latest version tested 3.14. 3
  • The graphviz graph layout engine (latest version tested: 2.28.0).
  • Pygraphviz, a python interface to graphviz (latest version tested: 1.1).
Python and Pyro are essential. PyGTK is required by the gcylc GUI (but you can control and monitor cylc suites from the command line). Graphviz and pygraphviz are required for dependency graphing and the graph-based suite control GUI (but you can also run cylc without them). Cylc has absorbed the following software in modified form (no need to install them):
  • xdot, a graph viewer (LGPL license)
  • the ConfigObj and Validate modules (BSD license)

Development

Cylc uses the powerful Git distributed source code management system. Git makes branching and merging, forking/cloning, and pushing and pulling changes between repositories very easy. The cylc master repository is hosted on GitHub.

At this stage we envision using the fork and pull model of collaborative development. Anyone can fork the cylc repository on GitHub (or just locally) and push changes to their personal fork (or just work in their local clone) without requiring write access to the master repository. The changes must then be pulled into the source repository, reviewed, and merged, by the project maintainer. Thus we avoid the overhead, and potential problems, of managing permissions for multiple users and many personal branches within the one repository, and without restricting development (because everyone gets their own fully functional cylc repository).

Ultimately, to reduce the burden of maintainance on the master repository, we could have a top tier of official forks that the project maintainer collaborates with directly, and others below, perhaps within specific cylc sites, that interact with the official collaborators. (But note that if circumstances warrant, anybody can easily and safely collaborate with anybody else using git, given read access to their respective repositories).
