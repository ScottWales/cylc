Cylc

Forecast Suite Metascheduler

Cylc ("silk") is an advanced metascheduler that makes it easy to construct and control flexible, portable suites for cycling weather and environmental forecasting systems, and related data processing. Cylc suites optimally interleave tasks from multiple forecast cycles for fast catch up from delays, fast case studies, and fast offset parallel test suites.

.

Provenance

Cylc is developed by Dr Hilary James Oliver at NIWA (the National Institute of Water and Atmospheric Research, New Zealand). It was prototyped in 2008, has controlled NIWA's multi-model operational forecasting systems since 2010, and was released under the GNU General Public License in 2011.

Overview

A metascheduler (in this context) is a control system that decides when members of a suite of inter-dependent tasks are ready to be submitted to a batch scheduler to be queued for execution. Cylc manages cycling (repeating) tasks related in terms of their respective forecast cycle times.

Dependence between tasks in different forecast cycles can usually be ignored in real time operation, but this severely restricts scheduling in delayed operation when it should be possible to run tasks from multiple cycles at once. Even simple suites become complicated when inter-cycle dependence is taken into account (in the illustrations a, b, and c are warm-cycled forecast models).

Cylc's self-organizing scheduling algorithm addresses this problem and does not use global cycling mechanisms to advance in time. This allows optimal interleaving of tasks from multiple forecast cycles, potentially achieving very fast catch up from delays even in resource limited operations with little down time between cycles (as shown here above the time line for the example of the previous paragraph).

Cylc also has very flexible, adaptive run time behaviour: suites organically transition between delayed and real time operation as required; they adapt to insertion and removal of tasks; and they flow around failed or delayed tasks - which catch up as quickly as possible once the problem is addressed.

Cylc's suite design paradigm is simple and intuitive: "draw" a dependency graph in concise textual form, then configure the execution environment and command line for each task. E.g.: the dependencies section in this suite definition defines this suite.

Other Notable Features

  • Tasks can be distributed across multiple hosts on a heterogeneous network, with multiple job submission methods.
  • Most existing models and processing jobs can be used without modification as cylc tasks; some require minimal modification.
  • Suite definitions are automatically validated against a comprehensive spec file, to catch most errors prior to run time.
  • Self-documenting unified command line interface covering all functionality from suite editing, searching, graphing, and validation; to interrogation and control.
  • Graphical interfaces covering the same, including a graph-based suite control interface. See screen shots.
  • Cryptographic suite security to prevent users from intervening in the operation of suites owned by others.
  • Run single tasks exactly as the suite would run them (job submission method and execution environment).
  • A network lockserver prevents accidental invocation of multiple instances of the same suite or task at the same time.
  • Suite databases enable sharing of suites among users.
  • Insertion and deletion of tasks at run time.
  • Explicit handling of cold start processing to bootstrap warm-cycling suites from scratch.
  • Suites can be cold started, warm started, or restarted from a previous state.
  • Suites can be stopped and restarted in any state of operation.
  • Restart states are automatically dumped prior to actioning any intervention in a running suite.
  • Spin-up processing by insertion of temporary tasks and addition of new prerequisites on the fly.
  • Recursive purge: remove an entire sub-tree of tasks through to some future cycle time.
  • Conditional prerequisites.
  • Task families.
  • Task retry for automated failure recovery.
  • Central alerting and timeouts for operational suites.
  • Portable suites can easily be constructed (no hardwired paths for suite control files or output directories, etc. - just copy a suite and run it).
  • Simulation mode: get the scheduling right, on an accelerated clock, without running real tasks.

Development

Cylc uses the powerful Git distributed source code management system. The master repository is here on GitHub.

At this stage the collaborative development model is envisioned as follows. Collaborators should:

  • Clone the cylc master repository on GitHub.
  • Develop new features in a new branch of their own repository.
  • Finally, merge in any new changes from the master repository, and,
  • Send a pull request to the master repository maintainer.
The master repository maintainer can then pull the new branch, review the changes, and merge into the main branch for future releases.

This model applies specifically to the top tier of cylc developers who need to interact directly with the master cylc repository. There can be lower tiers of development, within a particular organisation say, that work similarly below the top collaborator repositories.
