Cylc

Forecast Suite Metascheduler

Cylc ("silk") is an advanced metascheduler that makes it easy to construct and control flexible, portable suites for cycling weather and environmental forecasting systems, and related data processing. Cylc suites optimally interleave tasks from multiple forecast cycles for fast catch up from delays, fast case studies, and fast offset parallel test suites.

.

Provenance

Cylc is developed by Dr Hilary James Oliver at NIWA (National Institute of Water & Atmospheric Research, New Zealand). It was prototyped in 2008, has controlled NIWA's multi-model operational forecasting systems since 2010, and was released under the GNU General Public License in 2011.

Overview

A metascheduler controls the order in which members of a suite of inter-dependent tasks are executed (usually by submission to a batch queue scheduler) in order to avoid dependency violations. Cylc does this for suites of cycling tasks.

Dependence between tasks in different forecast cycles can usually be ignored in real time operation. This severely restricts scheduling after delays when it should be possible to run tasks from multiple cycles at once - but even simple suites become complicated when inter-cycle dependence is taken into account (in the illustrations a, b, and c represent warm-cycled forecast models).

Cylc solves this problem; it optimally interleaves tasks from multiple forecast cycles whenever the opportunity arises, potentially achieving very fast catch up from delays even in resource limited operations with little down time between cycles. The same occurs for processing historical data, or running parallel test suites that start behind the main operation.

Cylc also has very flexible, adaptive run time behaviour: suites naturally transition between delayed and real time operation as required; they adapt to insertion and removal of tasks; and they flow around failed or delayed tasks (which catch up as quickly as possible once the problem is addressed).

Cylc's suite design paradigm is simple and intuitive: "draw" a dependency graph in concise textual form, then configure the execution environment and command line for each task. E.g.: this simple suite definition results in this dependency graph.

Other Notable Features

  • Tasks can be distributed across multiple hosts on a heterogeneous network, with multiple job submission methods.
  • Most existing models and processing jobs can be used without modification as cylc tasks; some require minimal modification.
  • Suite definitions are automatically validated against a comprehensive spec file, to catch most errors prior to run time.
  • Self-documenting command line interface covering all functionality from suite editing to suite control.
  • Graphical interfaces covering the same, including a graph-based suite control interface. See screen shots.
  • Cryptographic suite security: users cannot intervene in the operation of suites owned by others.
  • Run single tasks exactly as the suite would run them (job submission method and execution environment).
  • Network lockservers prevent accidental invocation of multiple instances of the same suite or task at the same time.
  • Suite databases enable sharing of suites among users.
  • Insertion and deletion of tasks at run time.
  • Explicit cold start processing to bootstrap warm-cycling suites from scratch.
  • Suites can be cold started, warm started, or restarted from a previous state.
  • Suites can be stopped and restarted in any state of operation.
  • Restart states are automatically dumped prior to actioning any intervention in a running suite.
  • Spin-up processing by insertion of temporary tasks and addition of new prerequisites on the fly.
  • Recursive purge: remove an entire sub-tree of tasks through to some future cycle time.
  • Conditional prerequisites (triggers).
  • Task families.
  • Task retry for automated failure recovery.
  • Central alerting and timeouts for operational suites.
  • Portable suites are easy to construct - just copy another user's suite and run it.
  • Simulation mode: get the scheduling right, on an accelerated clock, without running real tasks.

Getting Cylc

Requirements

  • Operating System: Unix or Linux
  • The Python programming language
  • Pyro 3 (Python Remote Objects)
  • Graphviz (graph layout engine)
  • Pygraphviz (a Python graphviz wrapper)

Download

You can download the latest release from GitHub: cylc-3.0.5.tgz.

Installation

Unpack the release tarball and follow the installation instructions in the Cylc User Guide, located in the doc sub-directory.

Development

Cylc uses the powerful Git distributed source code management system. The cylc master repository is hosted on GitHub.

To develop a new feature (or fix a bug, etc.) collaborators should:

  1. Fork (clone) the cylc master repository on GitHub
  2. Clone a local copy of their cylc fork on GitHub
  3. Develop and test their changes in a new branch of their repository
  4. Merge in any new master repository changes and resolve any conflicts
  5. Push their changes back to their cylc fork on GitHub
  6. Send a pull request to Hilary Oliver
Hilary Oliver (the master repository maintainer) can then:
  1. fetch the new branch to the master repository
  2. review the changes
  3. merge them into the cylc trunk for future releases

This model applies specifically to a top tier of official collaborators. There can be lower tiers of development, e.g. within a particular organisation, that only interact directly with their associated cylc collaborator repository (although there is nothing to stop anybody from easily collaborating with anybody else, given network access and permissions, with git).
